# Android App — Functional Feature Spec (MVP)

## Overview

The Android app is a background SMS bridge. It has no messaging UI. Its sole purpose is to:

1. Maintain a persistent WebSocket connection to the SMS Gateway server
2. Forward incoming SMS messages to the server
3. Execute outbound SMS send commands received from the server
4. Wake itself via FCM when the WebSocket is idle and the server has a queued message

The app is configured by scanning a QR code generated by the server. No manual URL or key entry is required.

---

## Out of Scope (MVP)

- Messaging UI (thread list, compose, message history)
- Default SMS app behaviour
- MMS support
- Multi-device management UI
- Dual-SIM selection UI (SIM 0 used by default)
- Remote disconnect / revoke from app

---

## Minimum Requirements

- Android 14 (API 34) or higher
- Google Play Services (required for FCM)
- Camera permission (for QR code scanning)
- A physical SIM card

---

## Permissions

| Permission | Required | Purpose |
|---|---|---|
| `RECEIVE_SMS` | Yes | Receive incoming SMS via `SMS_RECEIVED` broadcast |
| `SEND_SMS` | Yes | Send outbound SMS on server command |
| `CAMERA` | Yes | Scan QR code during setup |
| `RECEIVE_BOOT_COMPLETED` | Yes | Restart BridgeService after device reboot |
| `FOREGROUND_SERVICE` | Yes | Keep BridgeService alive |
| `FOREGROUND_SERVICE_CONNECTED_DEVICE` | Yes | Android 14 foreground service type for persistent device connections |
| `WAKE_LOCK` | Yes | Hold `PARTIAL_WAKE_LOCK` to keep CPU active when screen is off |
| `REQUEST_IGNORE_BATTERY_OPTIMIZATIONS` | Yes | Prompt user to exempt app from Doze mode |
| `INTERNET` | Yes | WebSocket connection to gateway server |
| `READ_PHONE_STATE` | Optional | Detect SIM availability and phone number |

> **Known limitation:** `SMS_RECEIVED` is an ordered broadcast. If the device's default SMS app aborts the broadcast before our app receives it, incoming SMS will not be forwarded. In practice, Google Messages and Samsung Messages do not abort this broadcast. Recommend using the app on a dedicated gateway device.

## QR Code Configuration

### Format

The server generates a QR code encoding a JSON payload:

```json
{
  "url":  "wss://your-server.com:8765",
  "key":  "sk-bridge-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx",
  "name": "My SMS Gateway"
}
```

- `url` — WebSocket server URL (`wss://` scheme, includes host and port)
- `key` — API key for authentication (sent as `Authorization: Bearer <key>` on WebSocket connect)
- `name` — Suggested device label shown in the server's device registry (user can edit)

### Server endpoint

```
GET /setup/qr
Authorization: Bearer <api-key>
```

Returns a PNG QR code image encoding the above JSON. Accessed by the server operator via a browser or the server's admin UI.

### Security note

The QR code contains the API key in plaintext. Treat it like a password — do not screenshot it or share it. The server should invalidate and re-generate the QR code if the key is rotated.

## Background Services

### BridgeService (Foreground Service)

**Type:** `foregroundServiceType="connectedDevice"`

**Persistent notification:**
```
Agentic SMS Bridge
● Connected · wss://your-server.com
```
- Notification channel: `bridge_status`, importance LOW (no sound, no heads-up)
- Notification is non-dismissable while service is running
- Tapping the notification opens the Status Dashboard

**Lifecycle:**
- Started on: app launch, boot (via BootReceiver), FCM wake (via FcmService)
- Holds `PARTIAL_WAKE_LOCK` while running
- Runs indefinitely until user taps "Disconnect and reset"

**WebSocket connection state machine:**

```
IDLE
  │ startService()
  ▼
CONNECTING ──── timeout 10s / auth error ──► FAILED (retry after backoff)
  │
  │ connected
  ▼
CONNECTED ──── network drop / server close ──► RECONNECTING
  │                                                │
  │                                           backoff wait
  │                                                │
  │ ◄──────────────────────────────────────────────┘
  │
  │ stopService()
  ▼
IDLE
```

**Reconnection backoff:**

| Attempt | Wait |
|---|---|
| 1 | 2s |
| 2 | 4s |
| 3 | 8s |
| 4 | 16s |
| 5+ | 30s (cap) |

Resets to 0 on successful connection.

**On connect, BridgeService sends `device.hello`:**
```json
{
  "type":           "device.hello",
  "device_id":      "<uuid stable per install>",
  "device_name":    "My SMS Gateway",
  "android_version": 14,
  "fcm_token":      "<current FCM token>",
  "sim_count":      1
}
```

**Heartbeat:** WebSocket ping frame every 30 seconds via OkHttp `pingInterval`. If no pong received within 10 seconds, treat as disconnected and trigger reconnect.

**On `sms.send` received from server:**
1. Extract `to`, `body`, `id` from message
2. Call `SmsManager.sendTextMessage()`
3. Register sent and delivery `PendingIntent`
4. On sent intent received: send `sms.status { ref_id, status: "sent" }`
5. On delivery intent received: send `sms.status { ref_id, status: "delivered" }`
6. On failure: send `sms.status { ref_id, status: "failed", error: "..." }`

---

### SmsReceiver (BroadcastReceiver)

**Listens for:** `android.provider.Telephony.SMS_RECEIVED`

**Behaviour:**
1. `onReceive()` called by OS when SMS arrives
2. Call `goAsync()` immediately to extend the processing window beyond `onReceive()` return
3. Extract sender number and message body from PDU extras
4. If BridgeService is running and WebSocket is connected: forward immediately
5. If BridgeService is not running or WebSocket is disconnected: start BridgeService, buffer message, send once connected
6. Call `pendingResult.finish()` when forwarding is complete

**Concatenated (multi-part) SMS:** PDU extras may contain multiple parts. Reassemble into a single message body before forwarding.

---

### FcmService (FirebaseMessagingService)

**`onMessageReceived()`:**
- Check `data["action"] == "wake_connect"`
- Start BridgeService if not already running
- BridgeService connects WebSocket → server flushes pending outbound queue

**`onNewToken()`:**
- Store new FCM token in `EncryptedSharedPreferences`
- If BridgeService is running and WebSocket is connected, send updated `device.hello` with new token

---

### BootReceiver (BroadcastReceiver)

**Listens for:** `android.intent.action.BOOT_COMPLETED`

**Behaviour:** Start BridgeService via `startForegroundService()`. No other logic.

## Credential Storage

| Data | Storage |
|---|---|
| Server URL | `EncryptedSharedPreferences` |
| API key | `EncryptedSharedPreferences` |
| Device ID | `EncryptedSharedPreferences` (generated once on first run, stable for life of install) |
| Device name | `SharedPreferences` (not sensitive) |
| FCM token | `EncryptedSharedPreferences` |
| Activity counters | `SharedPreferences` |

The device ID is a random UUID generated on first launch. It is not tied to any hardware identifier.

## Error States

| Error | User-visible behaviour |
|---|---|
| Connection refused | Status Dashboard: red "Disconnected", reconnect in progress with backoff |
| Unauthorized (401) | Status Dashboard: red "Invalid API key — re-scan QR to reconnect" — no reconnect attempted |
| SSL certificate error | Status Dashboard: red "TLS error — check server certificate" — no reconnect attempted |
| RECEIVE_SMS revoked | Status Dashboard: amber "SMS receive permission denied" |
| SEND_SMS revoked | Status Dashboard: amber "SMS send permission denied" |
| Network unavailable | Status Dashboard: amber "Waiting for network…" — reconnects when network returns |
| SMS send failure | `sms.status { status: "failed", error: "<SmsManager result code>" }` sent to server |
| FCM unavailable (no Play Services) | FCM status indicator hidden from dashboard; app relies on persistent WebSocket only |

## Security

- API key stored in `EncryptedSharedPreferences` (AES-256, Android Keystore-backed)
- API key sent only over WSS (TLS encrypted transport)
- FCM payloads contain no SMS content — wake signal only
- TLS certificate validation uses Android's default trust store (standard CA chain)
- Self-signed server certificates require manual trust store configuration (not supported in MVP)
